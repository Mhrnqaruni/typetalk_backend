## Master Plan Review

### Overall Assessment
The master plan is structurally close to the locked source plan: the seven phases are reasonable in size, and the broad progression from foundation to auth, sync, billing, quota, and hardening is workable. However, it is not yet safe to approve because it does not fully cover the locked v1 scope, and it contains at least one real dependency error plus several mandatory execution requirements that are only mentioned indirectly or not assigned to any phase at all.

### Issues Found
- [HIGH] Incomplete project scope coverage. `master_plan.md:48-106` and `master_plan.md:123-133` never allocate work for several locked v1 endpoints required by `final_plan.md:130` and `final_plan.md:1038-1065`: `GET /v1/organizations/current`, `GET /v1/organizations/members`, `PATCH /v1/me`, `DELETE /v1/me`, `GET /v1/sessions`, `DELETE /v1/sessions/:sessionId`, `GET /v1/billing/plans`, and `GET /v1/billing/invoices`. This is wrong because the executor could follow the master plan exactly and still finish with required API surface missing. Fix by assigning each missing route to a specific phase with explicit deliverables and definition-of-done coverage. Verify by doing a route-by-route audit against the full API list in `final_plan.md` and requiring tests or endpoint checks for every listed route.
- [HIGH] Incorrect dependency ordering for security-event emission. `master_plan.md:58` requires "security event hooks for suspicious refresh reuse" in Phase 1, but `master_plan.md:127-133` does not add the `security_events` table until Phase 6. The source plan makes suspicious refresh reuse emit a `security_event` (`final_plan.md:821-827`) and places the table itself in the security/audit schema (`final_plan.md:737`, `final_plan.md:1485-1493`). This is wrong because Phase 1 currently depends on persistence that the plan does not create until five phases later. Fix by either moving `security_events` into Phase 1 or explicitly defining an earlier, durable interim mechanism and later migration path. Verify by ensuring the schema exists before the auth flow that must write to it, then test refresh-token reuse and confirm a persisted event record is created.
- [HIGH] Phase 0 does not preserve the source plan's mandatory early Railway validation. `final_plan.md:1377-1383` requires deploying the health-check skeleton to Railway immediately and validating Railway build, environment, healthcheck, and Postgres connectivity early. `master_plan.md:33-46` only plans local verification and ends with a merely "Railway-deployable skeleton." That is wrong because the whole point of Phase 0 is to flush out deployment drift before deeper implementation starts. Fix by adding explicit GitHub/Railway setup, actual Phase 0 deployment, and Railway-backed health verification to the deliverables and exit criteria. Verify by requiring a successful Railway deployment that answers `/health` while connected to the Railway Postgres service.
- [HIGH] Idempotency is not actually scheduled where the source plan requires it. `master_plan.md:80` creates `idempotency_keys`, but there are no explicit phase tasks applying idempotency to device registration (`master_plan.md:67`), Google verification/restore (`master_plan.md:98-99`), or dictation finalization (`master_plan.md:113-119`), even though `final_plan.md:569-574` and `final_plan.md:1153` require those exact uses. This is wrong because duplicate requests can otherwise double-register devices, replay provider restore flows, or double-spend quota. Fix by adding concrete deliverables and tests for idempotent handling on every required route, not just the table. Verify by issuing repeated requests with the same idempotency key and confirming there is no duplicate state change.
- [MEDIUM] Phase 0 omits the locked test-environment setup. The source plan requires `.env.test` and a separate `typetalk_test` database (`final_plan.md:1232`, `final_plan.md:1249-1255`), but `master_plan.md:33-46` only covers `.env.example` and general test scripts. This is wrong because the plan leaves the test path underspecified and risks tests running against the development database. Fix by adding `.env.test`, separate test DB setup, and test database usage to Phase 0 deliverables. Verify by running the test suite against `typetalk_test` and confirming it does not touch `typetalk_dev`.

### Suggestions
- Add a final "scope matrix" subsection mapping every required `/v1/...` route and every required table from `final_plan.md` to an owning phase.
- Tighten dependency wording so the plan cannot be interpreted as skipping Phase 2 before Phase 3, since the document says execution should follow the locked phase order exactly.
- Upgrade each phase definition of done from summary prose to verifiable checks: routes implemented, schema migrated, duplicate-request behavior tested, and deployment/test environment validated where required.

### Verdict
VERDICT: NEEDS_REVISION
