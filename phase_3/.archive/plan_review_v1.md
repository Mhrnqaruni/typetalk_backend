## Phase 3 Plan Review — Round 1
### Overall Assessment
The plan covers most of the locked Phase 3 scope and the sequencing is mostly sound: baseline verification, configuration and schema work, seeded plans, read routes, write routes, webhook receipt, retry execution, then full regression coverage. It also correctly reuses the Phase 2 idempotency and pagination infrastructure instead of reinventing it. I am not approving it yet because three required behaviors are still underspecified in ways that could let the executor ship incorrect billing state even if the checklist is followed.

### Issues Found
- [HIGH] The retry model has no recovery path for abandoned `processing` rows — `phase_3/plan.md` Steps 13 and 15 — The plan introduces the durable `received -> processing -> processed/failed` state machine and a `locked_at` field, but the retry executor is only told to scan `received` and `failed`. If the process crashes after marking an event `processing`, that row can remain stuck forever and Stripe state plus entitlements can drift permanently. Fix suggestion: explicitly require stale-lock recovery semantics, for example reclaiming `processing` rows whose `locked_at` is older than a bounded timeout by moving them back into retry eligibility or treating them as failed, and add a failure-injection test that proves the cron path recovers a crash after the row is locked.
- [MEDIUM] The locked `billing_overlap` entitlement rule is still not scheduled — `phase_3/plan.md` Steps 8, 9, 14, Testing Strategy, and Success Criteria — The final plan requires that if more than one paid subscription is active for the same organization, access stays paid and `billing_overlap` is set. Phase 3 blocks duplicate checkout creation, but it never explicitly requires entitlement recomputation to detect and persist overlap when Stripe already has multiple active subscriptions for the org. Fix suggestion: add overlap detection to the entitlement engine now, require `billing_overlap` to be set and cleared during Stripe event processing, and add a test that seeds two active Stripe subscriptions for one organization and verifies paid access plus `billing_overlap=true`.
- [MEDIUM] Webhook payload privacy and retention are missing from the implementation contract — `phase_3/plan.md` Steps 3, 13, and Testing Strategy — The final plan allows storing raw webhook payloads only short-term if sensitive and forbids storing raw payment instrument data by default. This plan adds durable `payload_json` storage but never says whether Phase 3 stores the full Stripe event forever, stores only a minimal verified snapshot, or prunes sensitive payloads after processing. Fix suggestion: explicitly require one safe behavior in the schema and processing steps, define the retention window if raw payloads are kept temporarily, and add verification that persisted webhook rows do not retain unnecessary sensitive Stripe objects indefinitely.

### Positive Aspects
- The plan covers the full Phase 3 route and table surface from the master plan, including the Railway-cron retry executor.
- It correctly reuses the existing current-organization, idempotency, and cursor-pagination extension points instead of duplicating infrastructure.
- The success criteria and test matrix are strong around checkout idempotency, raw-body signature verification, duplicate-event dedupe, and forced transient retry testing.

### Verdict
VERDICT: NEEDS_REVISION
