## Phase 4 — Implementation Plan

### Changes After Review Round 1

- Inspector comment: the plan never scheduled population of the DB-backed Google plan mapping even though the live schema and seed path already expose `plans.google_product_id` and `plans.google_base_plan_id`. What was the issue: I confirmed in `prisma/schema.prisma` that the columns already exist, and I confirmed in `prisma/seed.ts` that the current `pro_monthly` and `pro_yearly` seed rows still set both Google fields to `null`. The earlier Phase 4 plan never assigned work to populate those fields or to require plan resolution through the `plans` table. What I changed: I added a dedicated Step 5 for extending `prisma/seed.ts` and related test fixtures so `pro_monthly` and `pro_yearly` have non-null Google product and base-plan identifiers in the database, updated Step 7 and Step 9 so Google verify and restore resolve plans through those seeded rows instead of hardcoded service constants, and added matching testing and success-criteria requirements. Why: the locked backend design requires billing plan metadata to come from the database, and Google Play verification cannot safely map purchases to entitlements if the catalog-to-provider mapping is left implicit.
- Inspector comment: acknowledgment retry ownership was incomplete for failures that happen during `verify-subscription` or `restore` before any RTDN exists. What was the issue: I confirmed the earlier Step 9 only said the first acknowledgment failure must leave a retry path, while the earlier Step 15 only extended the `webhook_events` retry executor for RTDN-backed rows. That left a gap for "verify succeeds, acknowledgment fails, and no RTDN arrives." What I changed: Step 3 now requires durable acknowledgment-retry fields on `purchase_tokens`, Step 10 defines route-originated acknowledgment failure persistence on that table, Step 11 reuses the same durable acknowledgment path from restore, and Step 16 extends scheduled reconciliation so it scans both Google RTDN rows and pending `purchase_tokens` acknowledgment retries even when no RTDN has arrived. The testing strategy and success criteria now require the exact "verify succeeds, initial ack fails, no RTDN arrives, scheduled retry later acknowledges without duplicating entitlement grants" case. Why: acknowledgment recovery must not depend on receiving a later webhook, or valid Android purchases can stay unacknowledged indefinitely despite successful secure verification.
- Inspector comment: RTDN dedupe was underspecified because the plan never defined what Google value becomes `webhook_events.external_event_id`. What was the issue: I confirmed the Phase 3 schema already enforces unique `(provider, external_event_id)` on `webhook_events`, but the earlier Phase 4 plan never bound Google RTDN to a stable transport identifier. That could lead to deduping on purchase token or another business field that is shared across multiple legitimate lifecycle events. What I changed: Step 14 now requires the RTDN route to extract the stable Pub/Sub `messageId` from the delivery envelope, Step 15 now requires `webhook_events.external_event_id` for Google RTDN to be that transport message id rather than `purchaseToken`, and Step 16 plus the testing strategy now require separate tests for duplicate redelivery of the same Pub/Sub message versus distinct lifecycle messages concerning the same purchase token. Why: deduping on the transport identity preserves retry-safe redelivery handling without collapsing separate Google subscription events that happen to reference the same purchase token.

### Objective

Phase 4 extends the existing Stripe-first billing system to support Android subscriptions through secure Google Play verification, durable purchase-token storage, RTDN ingestion, and unified entitlement recomputation. At the end of this phase, Google Play and Stripe must both drive the same organization-scoped billing and entitlement APIs without granting access from untrusted or pending provider state.

### Prerequisites

- Phase 0, Phase 1, Phase 2, and Phase 3 must already be completed and re-verified in the current workspace.
- The current backend must still pass `npm run build` and `npm run test` before any Phase 4 edits begin.
- `typetalk_dev` and `typetalk_test` must already be migrated through the Phase 3 Stripe billing schema and seeded with the canonical plans.
- The existing shared primitives from earlier phases must remain the extension points for this work: current-organization resolution, actor-scoped idempotency, shared cursor pagination, unified entitlement recomputation, and the Phase 3 durable `webhook_events` retry pipeline.
- Google Play configuration required by the locked final plan must be available in local and test env files before live-provider integration is attempted: package name, service-account credentials or key path, any issuer/client metadata needed for Android Publisher API access, RTDN trust-verification settings, and bounded retry/lock settings for the extended reconciliation executor.
- Phase 4 must not implement Phase 5 usage metering or quota enforcement logic; this phase is limited to Google Play billing, provider sync, RTDN durability, and unified billing/entitlement reads.
