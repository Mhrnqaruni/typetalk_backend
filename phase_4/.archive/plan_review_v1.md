## Phase 4 Plan Review — Round 1
### Overall Assessment
The plan is close and the overall phase shape is correct: it keeps Google Play work on top of the existing Phase 3 billing/entitlement stack, covers verify, restore, RTDN, linked purchase tokens, overlap handling, and the shared retry pipeline, and it stays out of Phase 5 usage scope. I do not approve it yet because two production-critical contracts are still underspecified and one durable-webhook detail is still too loose for a billing phase.

### Issues Found
- [HIGH] The plan never schedules population of the DB-backed Google plan mapping, even though the live Phase 3 catalog already has `plans.google_product_id` and `plans.google_base_plan_id` columns and the current seed rows leave them `null` — where: current code in `prisma/schema.prisma` and `prisma/seed.ts`, and missing from Phase 4 Steps 2-5 / Success Criteria — fix suggestion: add an explicit Phase 4 step to extend `prisma/seed.ts` and related config/test fixtures so `pro_monthly` and `pro_yearly` are mapped to real Google product/base-plan identifiers in the database, then require tests that verify Google verify/restore/RTDN resolve plans through the `plans` table instead of hardcoded service constants.
- [HIGH] Acknowledgment retry ownership is still incomplete for failures that happen during `verify-subscription` or `restore` before any RTDN exists. Step 9 says the first acknowledgment failure must leave a retry path, but Step 15 only extends the existing `webhook_events` retry executor, which scans RTDN-backed events and does not inherently cover a client-route acknowledgment failure — where: Phase 4 Steps 9 and 15, Testing Strategy, and Success Criteria — fix suggestion: explicitly define the durable retry carrier for verify/restore-originated ack failures, such as retry fields on `purchase_tokens` plus scheduled reconciliation, or insertion of a retryable synthetic job/event row when ack fails. Add a required test for "verify succeeds, initial ack fails, no RTDN arrives, scheduled retry later acknowledges without duplicating entitlement grants."
- [MEDIUM] RTDN dedupe is not concrete because the plan never defines what Google value becomes `webhook_events.external_event_id`. The Phase 3 schema already enforces unique `(provider, external_event_id)`, so leaving the Google key implicit risks deduping on purchase token or another non-transport field, which can collapse distinct lifecycle events or fail to dedupe redeliveries — where: current Phase 3 `webhook_events` schema in `prisma/schema.prisma`, and Phase 4 Steps 13-15 — fix suggestion: explicitly bind Google RTDN dedupe to the stable Pub/Sub transport identifier that survives redelivery, keep purchase token and notification type in the payload/sync state instead, and require tests that distinguish duplicate redelivery of the same message from separate events concerning the same purchase token.

### Positive Aspects
- The phase scope is aligned with the locked final plan: Google verify, restore, RTDN, secure provider sync, acknowledgment, overlap handling, and unified read APIs are all present.
- The technical direction is sound in that it extends the existing Phase 3 abstractions instead of proposing a parallel Google-only billing stack.
- The plan is concrete about pending-state denial, linked-token handling, stale-lock recovery, and regression coverage against the existing Stripe paths.

### Verdict
VERDICT: NEEDS_REVISION
