## Phase 2 Plan Review - Round 1

### Overall Assessment
I re-read the locked backend plan, the master plan, the current project status, the current Phase 1 execution report, and the actual Phase 1 code that Phase 2 is supposed to build on. The Phase 2 plan is mostly well structured: it covers the full route and table scope from the master plan, it correctly recognizes that device state already lives inside the auth flow today, and it includes a serious testing section that keeps Phase 1 regressions in scope.

I do not approve it yet. The main problem is that this is the first phase that introduces `idempotency_keys`, but the plan still leaves the user-owned idempotency contract too implicit. Two additional API contracts are also not concrete enough yet: the exact default preferences response and the actual limits for "bounded" JSON profile payloads.

### Issues Found
- [HIGH] Idempotency is still under-specified for an authenticated user-owned write path - `phase_2/plan.md:25`, `phase_2/plan.md:37`, `phase_2/plan.md:39`, `phase_2/plan.md:49`, `phase_2/plan.md:51` - the plan says same key plus same hash should replay and same key plus different hash should conflict, but it never locks the namespace and atomicity rules that make that safe. It does not explicitly require actor-scoped uniqueness for `scope`, so the executor could implement a global key space for `POST /v1/devices/register` and create false conflicts or even stored-response replay across users. It also does not explicitly require the idempotency-key claim, business write, and stored-response persistence to happen in one transaction or explicit claim/finalize state machine. Fix suggestion: explicitly require `(scope, idempotency_key)` to be unique with `scope` including at least route plus authenticated user id (and organization when relevant), and require one transactional/idempotent state-machine flow that claims the key, performs the device write, stores the response, and replays only actor-owned results. Verify with tests for concurrent duplicate requests by the same user and for two different users reusing the same external idempotency key.
- [MEDIUM] The preference-default contract is not measurable yet - `phase_2/plan.md:73`, `phase_2/plan.md:74`, `phase_2/plan.md:75`, `phase_2/plan.md:124`, `phase_2/plan.md:125` - the plan requires `GET /v1/preferences` to return "stable defaults" before the first write, but it never defines what those defaults are for `default_language`, `auto_punctuation`, `remove_fillers`, and `auto_format`. It also does not say whether `PUT /v1/preferences` is full replacement or partial merge when fields are omitted. That leaves room for inconsistent implementations while still claiming the plan was followed. Fix suggestion: lock the exact default values and the exact `PUT` contract now, including whether every field is required and what omitted fields mean. Verify with a brand-new user on two devices and with request-shape tests that prove the update semantics are deterministic.
- [MEDIUM] "Bounded" JSON validation for synced profiles is too vague - `phase_2/plan.md:37`, `phase_2/plan.md:91`, `phase_2/plan.md:94`, `phase_2/plan.md:97`, `phase_2/plan.md:99` - the plan repeatedly requires bounded validation for `rules_json` and `settings_json`, but it never defines the actual bounds. Without concrete limits on top-level type, maximum size, nesting depth, or key count, the executor can accept arbitrarily large or inconsistent payloads while still claiming they are "bounded." Fix suggestion: define explicit validation rules now, such as object-only payloads, maximum serialized size, maximum nesting depth, and any reserved/forbidden field patterns. Verify with boundary tests that cover both accepted and rejected payload sizes and shapes.

### Positive Aspects
- The plan covers every Phase 2 table and route assigned by the master plan, including `idempotency_keys`, all four device routes, preferences, dictionary, writing profiles, and app profiles.
- The ordering is mostly sound: schema first, migration second, shared infrastructure before routes, and final full-suite verification after module registration.
- The plan correctly uses the actual current baseline instead of pretending devices are not already handled by Phase 1 auth flows.
- Ownership, pagination, cross-device sync behavior, and session cleanup on device deletion are all explicitly called out, which reduces the risk of shallow happy-path implementation.
- The testing strategy is strong and appropriately insists that Phase 1 auth tests still pass after the device extraction.

### Verdict
VERDICT: NEEDS_REVISION
