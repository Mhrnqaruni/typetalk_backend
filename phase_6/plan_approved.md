## Phase 6 — Implementation Plan

### Changes After Review Round 2
- Inspector issue: the Round 1 transition policy still left `POST /v1/billing/stripe/checkout-session` callable, which in the current codebase is still a live Stripe write path that creates new checkout sessions. What I changed: tightened the Phase 6 transition policy so Stripe checkout is no longer an allowed launch write path after Paddle becomes default; the plan now says Stripe checkout must be retired from the public launch surface or hard-gated to reject normal launch traffic without creating provider side effects, while Stripe customer-portal, webhook, retry, and historical read flows remain available only for organizations that already have Stripe-backed subscriptions until cleanup is approved. Why: leaving Stripe checkout callable would allow new Stripe subscriptions after the migration and keep the web/Windows provider boundary ambiguous.
- Inspector issue: Step 3 still omitted the concrete `BillingService` constructor/injection work needed to make Paddle a first-class provider. What I changed: added `src/modules/billing/service.ts` to the provider-wiring step and made the dependency decision explicit: `BillingService` must move from separate `stripeProvider`/`googlePlayProvider` constructor fields to a provider-aware injected dependency bundle or equivalent orchestration boundary that can carry Paddle, Google Play, and legacy Stripe cleanly through app bootstrap and tests. Why: the current constructor shape is the actual bottleneck for injecting Paddle through `src/app.ts` and `test/helpers/app.ts`.

### Changes After Review Round 1
- Inspector issue: the legacy Stripe transition-policy decision happened too late, after multiple steps that depended on it. What I changed: moved the transition-policy decision to Step 1 and made it concrete: `stripe_price_id` remains only as nullable legacy plan metadata during Phase 6, legacy Stripe transitional behavior remains explicit instead of implicit, Stripe webhook retry stays enabled while historical Stripe events remain processable, and historical Stripe subscriptions/events remain readable in billing and entitlement flows while all new web/Windows billing actions move to Paddle. Why: test expectations, schema updates, seed/catalog changes, retry logic, and provider-precedence work all depend on those rules.
- Inspector issue: the plan-catalog step did not include the service/test work needed for `GET /v1/billing/plans` to actually become Paddle-first. What I changed: expanded the plan-catalog step to include `src/modules/billing/service.ts` and the billing integration tests, and added an explicit API decision that `GET /v1/billing/plans` must serialize Paddle launch metadata first while exposing Stripe identifiers only as optional legacy metadata during the transition. Why: changing only the seed and repository lookups would not change the HTTP response contract.

### Objective
Migrate web and Windows billing from the current historical Stripe-first implementation to a Paddle-first launch model while keeping Android on Google Play and preserving correct unified entitlements. Phase 6 must leave Paddle as the active web/Windows provider, keep legacy Stripe handling explicit and safe during transition, and preserve the already-approved Phase 4 Google Play and Phase 5 usage/quota behavior.

### Prerequisites
- Phase 5 must be fully approved and the current baseline must still pass `npm run build` and `npm run test`.
- Local development and test databases must be up to date with all migrations through Phase 5, and `prisma/seed.ts` must still seed the current plan catalog successfully.
- Mandatory repository and deployment readiness checks must pass before implementation starts: `git rev-parse --is-inside-work-tree`, `git remote -v`, `gh auth status`, `git ls-remote https://github.com/Mhrnqaruni/typetalk_backend.git`, `railway whoami`, and `railway status`.
- A GitHub backup checkpoint must be created before any Phase 6 schema or billing-provider changes.
- Paddle sandbox credentials and identifiers must be available before coding starts: API key, webhook secret, environment mode, monthly plan identifier, yearly plan identifier, and whatever metadata is needed for Paddle checkout/self-service flows.
- The current Stripe-first billing files must be treated as the migration baseline, not rewritten blindly: `src/modules/billing/provider.ts`, `src/modules/billing/service.ts`, `src/modules/billing/routes.ts`, `src/modules/billing/repository.ts`, `src/modules/billing/schemas.ts`, `src/modules/billing/stripe.ts`, `src/modules/billing/google-play-support.ts`, `src/modules/entitlements/service.ts`, `src/app.ts`, `src/jobs/webhook-retry.ts`, `src/config/env.ts`, `prisma/schema.prisma`, `prisma/seed.ts`, and the current billing integration tests.

### Steps
1. Lock the Stripe-to-Paddle transition policy before touching tests, schema, or provider logic.
   What to do: Define the exact migration rules up front and treat them as implementation constraints for the rest of the phase. The policy for Phase 6 must be: `plans.paddle_price_id` becomes the active web/Windows launch identifier; `plans.stripe_price_id` remains only as nullable legacy metadata during transition; all new web/Windows checkout and self-service actions move to Paddle; `POST /v1/billing/stripe/checkout-session` is no longer a launch write path and must be either removed from the public launch surface or hard-gated to reject normal requests before any Stripe session is created; any remaining Stripe customer-portal behavior is only for organizations that already have Stripe-backed subscriptions/customers; Stripe webhook retry remains enabled while historical Stripe events exist; and historical Stripe subscriptions/events remain readable and entitlement-resolvable during the migration.
   Which files are affected: `src/modules/billing/service.ts`, `src/modules/billing/routes.ts`, `src/modules/billing/repository.ts`, `src/jobs/webhook-retry.ts`, `prisma/schema.prisma`, `prisma/seed.ts`, `test/integration/billing.stripe.test.ts`, `test/integration/billing.webhooks.test.ts`, and the new Paddle-focused tests added in the next step.
   Expected outcome / how to verify: The migration rules are explicit before any implementation work branches on provider behavior. During execution review, every later step should be traceable back to these policy rules, and no later step should still be deciding whether Stripe checkout is writable, whether Stripe customer-portal access is limited to existing Stripe organizations, or whether Stripe is launch-default.
   Potential risks: If these rules are not locked first, later changes can silently keep Stripe as the de facto default, keep new Stripe checkout writes alive after the migration, or break historical Stripe readability/retryability while Paddle is being introduced.

2. Encode the Phase 6 target behavior and transition rules in focused integration coverage before refactoring.
   What to do: Add or update focused billing integration tests so the desired Phase 6 behavior is executable and measurable. The coverage should include Paddle checkout, Paddle customer self-service, Paddle webhook receipt/processing, Paddle retry behavior, Google Play non-regression, proof that the Stripe checkout route is disabled or hard-gated exactly as documented, proof that any remaining Stripe customer-portal behavior only works for existing Stripe-backed organizations, historical Stripe read/entitlement continuity, and the updated `GET /v1/billing/plans` contract.
   Which files are affected: `test/helpers/app.ts`, `test/integration/billing.stripe.test.ts`, `test/integration/billing.webhooks.test.ts`, `test/integration/billing.google-play.test.ts`, `test/integration/billing.google-rtdn.test.ts`, and new focused Paddle tests such as `test/integration/billing.paddle.test.ts` and `test/integration/billing.paddle-webhooks.test.ts`.
   Expected outcome / how to verify: The Phase 6 target state is test-backed before the billing implementation is refactored. New Paddle-focused tests should initially fail for missing implementation, while Google Play and legacy Stripe tests document the transition policy precisely, including a negative test proving Stripe checkout no longer creates a new checkout session after the migration behavior is in place.
   Potential risks: If the tests are written before the transition rules are locked, the suite can encode contradictory expectations for launch-default behavior versus legacy transitional behavior.

3. Extend the provider abstraction, live adapters, and test harness so Paddle is a first-class provider.
   What to do: Introduce Paddle-specific provider types and a live Paddle adapter while preserving the existing Stripe and Google Play integrations long enough for the transition. Change `BillingService` from its current separate `stripeProvider` plus `googlePlayProvider` constructor shape to a provider-aware injected dependency bundle or equivalent orchestration boundary that can carry Paddle, Google Play, and legacy Stripe cleanly. Then update the app bootstrap, retry job construction, and test harness to instantiate BillingService through that new dependency shape in the combinations required by the new tests.
   Which files are affected: `src/modules/billing/provider.ts`, a new Paddle adapter file such as `src/modules/billing/paddle.ts`, `src/modules/billing/service.ts`, `src/app.ts`, `src/jobs/webhook-retry.ts`, and `test/helpers/app.ts`.
   Expected outcome / how to verify: The codebase can build and test against Paddle-backed web/Windows flows without removing Google Play or legacy Stripe support too early. `npm run build` should still pass after the provider interface and wiring changes.
   Potential risks: The current abstraction is Stripe-shaped, so over-generalizing it can create churn or hidden regressions in the already-working Google Play and Stripe code paths.

4. Make Paddle configuration first-class and move Stripe configuration to explicit legacy support.
   What to do: Update environment loading and validation so Paddle secrets and identifiers are the primary web/Windows billing configuration, while current Stripe variables are retained only as explicit legacy inputs during the migration window. Keep local and test env templates aligned so the code can boot in development, test, and Railway without hidden config drift.
   Which files are affected: `src/config/env.ts`, `.env.example`, `.env.test`, and any existing tracked environment templates needed for local/test startup.
   Expected outcome / how to verify: Environment parsing succeeds with Paddle-first config, the app/test harness can boot with Paddle placeholder values in non-production contexts, and the configuration surface clearly distinguishes active Paddle launch settings from legacy Stripe settings.
   Potential risks: Making Paddle values mandatory too early can break local/test boot; leaving Stripe mandatory and undocumented as legacy can keep the deployment model misaligned with the locked launch plan.

5. Update the Prisma billing schema and migration so Paddle is represented directly without losing historical Stripe context.
   What to do: Modify the billing schema so `BillingProvider` includes `PADDLE`, plan catalog rows can store Paddle identifiers directly, and legacy Stripe identifiers remain readable only as transitional metadata. Add a Phase 6 migration that safely updates enums, plan fields, indexes, and any minimal transitional billing metadata required for mixed-provider reads and retries.
   Which files are affected: `prisma/schema.prisma` and a new migration under `prisma/migrations/`.
   Expected outcome / how to verify: `npx prisma validate`, `npx prisma generate`, `npx prisma migrate status`, and explicit test-database `npx prisma migrate deploy` all succeed, and the migrated schema still supports existing Phase 3/Phase 4 rows while enabling Paddle-backed plan and provider lookups.
   Potential risks: Enum changes can be migration-sensitive, new uniqueness constraints can conflict with existing seed data, and a careless schema change can make historical Stripe subscriptions or webhook rows unreadable.

6. Update the plan catalog, billing repository lookups, and `/v1/billing/plans` response so the API becomes Paddle-first.
   What to do: Replace Stripe-first plan seeding with Paddle-first plan seeding for the web/Windows paid plans while keeping Google Play identifiers intact and retaining Stripe identifiers only when the transition policy requires them for historical interpretation. Update repository plan-resolution methods for Paddle identifiers, and update `BillingService.listPlans()` plus the relevant integration tests so `GET /v1/billing/plans` explicitly returns Paddle launch metadata first and only includes Stripe identifiers as optional legacy metadata during the transition.
   Which files are affected: `prisma/seed.ts`, `src/modules/billing/repository.ts`, `src/modules/billing/service.ts`, `test/integration/billing.paddle.test.ts`, and `test/integration/billing.stripe.test.ts`.
   Expected outcome / how to verify: `npx prisma db seed` writes the correct Paddle-first plan catalog, `GET /v1/billing/plans` reflects Paddle launch metadata instead of Stripe-first metadata, and repository-level billing operations can resolve plans for Paddle and Google Play without relying on `stripe_price_id` as the active paid-plan source of truth.
   Potential risks: Seeding over existing rows can accidentally erase historical Stripe context, and changing the plans API without synchronized tests can leave the HTTP contract inconsistent with the underlying catalog.

7. Replace the active web/Windows checkout and self-service routes with Paddle-backed behavior.
   What to do: Implement `POST /v1/billing/paddle/checkout` and `POST /v1/billing/paddle/customer-portal` or equivalent self-service, and switch the service logic so new web/Windows billing actions use Paddle. In the same step, retire `POST /v1/billing/stripe/checkout-session` from normal launch traffic by either removing it from the public route surface or hard-gating it to return the documented non-launch error before any Stripe checkout session is created. If `POST /v1/billing/stripe/customer-portal` remains during transition, it must only work for organizations with existing Stripe-backed customers/subscriptions and must not be presented as the launch-default self-service path.
   Which files are affected: `src/modules/billing/routes.ts`, `src/modules/billing/service.ts`, `src/modules/billing/schemas.ts`, `src/modules/billing/provider.ts`, and the new Paddle adapter file.
   Expected outcome / how to verify: The new Paddle routes create checkout/self-service sessions idempotently, honor trial rules, and still block duplicate paid checkout when active paid access already exists. The new Paddle checkout/self-service tests should pass, and legacy Stripe route tests should prove that Stripe checkout is disabled or hard-gated exactly as documented and that any remaining Stripe customer-portal flow is limited to existing Stripe-backed organizations.
   Potential risks: Paddle's checkout/self-service model may not map one-to-one with the current Stripe assumptions around customer creation, trial handling, or return URLs, and an incomplete Stripe checkout gate could still leave a hidden write path active.

8. Implement Paddle webhook verification, durable receipt, deduplication, processing, and entitlement recomputation.
   What to do: Add the Paddle webhook endpoint and service flow using the same durable principles already required elsewhere: verify authenticity from the raw request body, insert the webhook row first, dedupe by external event id, serialize a minimal safe snapshot, process through a real provider-aware state machine, and recompute entitlements from the resulting subscription state.
   Which files are affected: `src/modules/billing/routes.ts`, `src/modules/billing/service.ts`, `src/modules/billing/repository.ts`, `src/modules/billing/provider.ts`, the new Paddle adapter file, and the focused Paddle webhook tests.
   Expected outcome / how to verify: Invalid Paddle webhook signatures are rejected, valid webhooks persist a `webhook_events` row before processing, duplicate deliveries do not create duplicate side effects, and successful processing updates `subscriptions` and `entitlements` correctly.
   Potential risks: Mapping Paddle event types and statuses into the existing subscription/entitlement model can be error-prone, especially around trials, cancellations, payment issues, and out-of-order deliveries.

9. Extend the shared webhook retry job so Paddle events are retry-safe without regressing Google RTDN or legacy Stripe retries.
   What to do: Update the retry job and webhook-processing logic so Paddle `webhook_events` rows can be claimed, processed, retried after transient failures, and reclaimed from stale locks. Keep Google RTDN retry behavior intact, and keep Stripe retry enabled as long as the transition policy says historical Stripe events remain processable.
   Which files are affected: `src/jobs/webhook-retry.ts`, `src/modules/billing/service.ts`, `src/modules/billing/repository.ts`, `test/integration/billing.paddle-webhooks.test.ts`, `test/integration/billing.webhooks.test.ts`, and `test/integration/billing.google-rtdn.test.ts`.
   Expected outcome / how to verify: A failed Paddle webhook row can be retried successfully by the job, stale Paddle `PROCESSING` locks can be reclaimed, Google RTDN retry tests still pass unchanged, and legacy Stripe retry tests still pass if Stripe retry remains enabled during transition.
   Potential risks: Provider-specific retry logic can drift, stale-lock handling can become inconsistent across providers, and shared retry changes can break already-correct Google RTDN behavior.

10. Update unified billing reads, invoice sourcing, and entitlement precedence for mixed Paddle, Google Play, and legacy Stripe state.
    What to do: Refactor billing summary, invoices, and entitlement recomputation so the system cleanly handles Paddle, Google Play, and explicit transitional Stripe records in the same organization. This includes choosing the correct primary subscription, surfacing overlap when multiple active paid providers exist, and making invoice reads provider-aware under the transition policy.
    Which files are affected: `src/modules/billing/service.ts`, `src/modules/entitlements/service.ts`, `src/modules/billing/subscription-access.ts`, `src/modules/billing/google-play-support.ts`, and any related billing tests that assert summary or invoice behavior.
    Expected outcome / how to verify: `GET /v1/billing/subscription`, `GET /v1/billing/invoices`, and `GET /v1/entitlements/current` return correct results for Paddle-only, Google-only, mixed-provider overlap, and legacy Stripe historical cases. Focused integration tests prove provider precedence, overlap flags, invoice-source selection, and source-provider reporting.
    Potential risks: Primary-subscription selection can choose the wrong provider, invoices can silently fall back to the wrong source, and mixed-provider organizations can become difficult to reason about without explicit scenario coverage.

11. Finalize explicit legacy Stripe behavior and rerun the full Phase 6 verification matrix plus deployment-readiness checks.
    What to do: Ensure the final code clearly distinguishes Paddle launch-default behavior from temporary legacy Stripe support, then run the full verification matrix: schema validation, generation, migration checks, seed, focused Paddle tests, focused Google Play non-regression tests, retry tests, build, and the full regression suite. Re-run the tracked-repo and Railway readiness checks, and confirm `.env.example`, startup wiring, and retry wiring match Paddle-first launch behavior.
    Which files are affected: No source files should change if all verification passes; if readiness mismatches are found, the likely corrections are in `src/config/env.ts`, `.env.example`, `src/app.ts`, `src/jobs/webhook-retry.ts`, `src/modules/billing/service.ts`, or the relevant billing tests.
    Expected outcome / how to verify: `npx prisma validate`, `npx prisma generate`, `npx prisma migrate status`, test-database `npx prisma migrate deploy`, `npx prisma db seed`, `npm run build`, targeted Phase 6 billing tests, Google Play billing tests, retry tests, and `npm run test` all pass. `git rev-parse`, `git remote -v`, `gh auth status`, `git ls-remote`, `railway whoami`, and `railway status` also pass so the migration is ready for execution review and later deployment work.
    Potential risks: Hidden mixed-provider regressions can remain green in narrow tests, GitHub backup may still be blocked if `gh auth status` is not repaired, and Railway env drift can be missed if only local verification is performed.

### Testing Strategy
- Run schema and migration validation first: `npx prisma validate`, `npx prisma generate`, `npx prisma migrate status`, and explicit test-database `npx prisma migrate deploy`.
- Reseed after schema/catalog changes with `npx prisma db seed` and verify the plan catalog now contains Paddle-first launch identifiers while keeping Google Play identifiers intact and any legacy Stripe metadata only where the transition policy allows it.
- Add focused integration coverage for:
  - Paddle checkout idempotency, trial handling, and duplicate-paid-checkout blocking
  - Paddle customer self-service route behavior
  - Paddle webhook verification, durable insert-first receipt, deduplication, successful processing, and failure/retry recovery
  - explicit `GET /v1/billing/plans` response expectations for Paddle-first metadata and optional legacy Stripe metadata
  - Google Play verify/restore/RTDN non-regression after the Paddle refactor
  - unified entitlement and billing-summary behavior for Paddle-only, Google-only, overlap, and legacy Stripe historical cases
  - explicit transitional Stripe behavior: Stripe checkout is disabled or hard-gated with no provider side effect, any remaining Stripe customer-portal path is limited to existing Stripe-backed organizations, and Stripe webhook/retry behavior continues only for historical events if Stripe remains temporarily supported
- Run the focused billing suites before the full suite, then run `npm run build` and `npm run test`.
- Use at least one end-to-end smoke flow per provider path:
  - Paddle checkout -> webhook -> subscription summary -> entitlements
  - Google Play verify -> billing summary -> entitlements
  - failed Paddle webhook -> retry job -> successful entitlement update
  - legacy Stripe historical row -> read-side billing/entitlement continuity if transitional Stripe support is still enabled
- Re-run the mandatory repo/deployment checks after the code passes locally: `git rev-parse --is-inside-work-tree`, `git remote -v`, `gh auth status`, `git ls-remote https://github.com/Mhrnqaruni/typetalk_backend.git`, `railway whoami`, and `railway status`.

### Success Criteria
- The Stripe-to-Paddle transition policy is explicit in the implementation: Paddle is the active web/Windows launch provider, new Stripe checkout writes are retired or hard-gated with no provider side effects, any remaining Stripe customer-portal behavior is restricted to existing Stripe-backed organizations, Stripe webhook retry remains enabled only as long as historical Stripe events still need support, and historical Stripe subscriptions/events remain readable during the migration.
- `BillingProvider` and related billing schema/configuration support Paddle as the active web/Windows launch provider while preserving Google Play and explicit historical Stripe handling.
- `GET /v1/billing/plans` returns Paddle-first launch metadata, and any remaining Stripe identifier fields are clearly optional legacy metadata rather than the active launch source of truth.
- `POST /v1/billing/paddle/checkout`, `POST /v1/billing/paddle/customer-portal`, and `POST /v1/webhooks/paddle` exist and work end to end.
- `GET /v1/billing/subscription`, `GET /v1/billing/invoices`, and `GET /v1/entitlements/current` reflect Paddle-first launch behavior without regressing Google Play support or historical Stripe readability.
- Paddle webhook events are verified from raw body, inserted durably before processing, deduplicated safely, and retried successfully through the shared webhook retry job.
- Google Play verify, restore, RTDN, acknowledgment retry, and entitlement recomputation still pass after the Paddle migration.
- The full verification matrix passes, and the tracked repository/GitHub/Railway readiness checks are green before Phase 6 is considered complete.
